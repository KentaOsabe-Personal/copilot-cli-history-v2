# Requirements Document

## Introduction

`cwd-session-tabs` は、GitHub Copilot CLI のローカル会話履歴を読み返す利用者が、現在の日付範囲と検索語に一致したセッション集合を作業ディレクトリ別に切り替えて確認できるようにする機能である。

この機能は、既存のセッション一覧 API が返す `work_context.cwd` を利用し、一覧画面の成功表示内でローカルに表示対象を切り替える。日付範囲、検索語、同期状態、一覧 API の契約は変更せず、複数プロジェクトの履歴が同じ一覧に混在する場合の探索負荷を下げる。

## Boundary Context

- **In scope**: セッション一覧の成功表示に `すべて`、作業ディレクトリ別、`ディレクトリ未設定` のタブを追加し、現在取得済みのセッション集合をタブ選択で切り替えること。
- **In scope**: `cwd` の未設定値、空白値、長いパス、タブ主ラベルを最後のディレクトリ名にする表示、同一 basename のパス、件数表示、選択中タブの補正、キーボード操作、アクセシビリティ名を利用者が判別できる形で扱うこと。
- **In scope**: 検索 UI と一覧成功時の説明文を、作業ディレクトリは一覧タブで切り替える文脈に寄せること。
- **Out of scope**: backend endpoint、DB schema、一覧 API response shape、server-side aggregation、directory summary API、pagination、repository / branch / model 専用フィルタ、並び替え UI、検索結果スコアリング、検索語ハイライトを追加または変更すること。
- **Out of scope**: backend の既存 `cwd` 検索互換性を削除すること、raw files から新しい cwd 推測値を作ること、過去 spec を書き換えること。
- **Adjacent expectations**: 日付範囲、検索語、履歴同期、一覧取得、カード表示は既存機能の契約を維持し、この機能はそれらが返したセッション集合を表示単位で切り替える。

## Requirements

### Requirement 1: 作業ディレクトリ別タブの提供

**Objective:** As a ローカル履歴を読み返す利用者, I want 現在の表示条件に一致したセッションを作業ディレクトリ別に切り替えたい, so that 複数プロジェクトの履歴が混在していても目的のセッションを見つけやすい

#### Acceptance Criteria

1. When セッション一覧画面が成功状態のセッション集合を表示する, the セッション一覧画面 shall `すべて` タブと作業ディレクトリ別タブを表示する
2. When 利用者が `すべて` タブを選択する, the セッション一覧画面 shall 現在の表示条件に一致した全セッションを既存の一覧順で表示する
3. When 利用者が作業ディレクトリ別タブを選択する, the セッション一覧画面 shall そのタブの作業ディレクトリに一致するセッションだけを既存の相対順で表示する
4. If セッションの `cwd` が `null` または空白である, then the セッション一覧画面 shall そのセッションを落とさず `ディレクトリ未設定` タブに含める
5. The セッション一覧画面 shall 各タブに該当セッション数を利用者が確認できる形で表示する

### Requirement 2: タブの表示名と識別性

**Objective:** As a 複数の作業ディレクトリをまたいで履歴を探す利用者, I want タブ名から対象ディレクトリを識別したい, so that 似た名前や長いパスでも誤ったディレクトリを選びにくい

#### Acceptance Criteria

1. When セッション集合に前後空白を含む `cwd` が存在する, the セッション一覧画面 shall 空白を除いた値で同じ作業ディレクトリとして扱う
2. When 複数セッションが同じ正規化済み `cwd` を持つ, the セッション一覧画面 shall それらを 1 つの作業ディレクトリ別タブにまとめる
3. When 作業ディレクトリのパスが存在する, the セッション一覧画面 shall タブの主ラベルに絶対パスではなく最後のディレクトリ名を表示し、完全なパスを確認できる情報も提供する
4. When 複数の作業ディレクトリが同じ basename を持つ, the セッション一覧画面 shall 利用者が各タブを区別できる追加のパス文脈を表示または提供する
5. The セッション一覧画面 shall `ディレクトリ未設定` タブに未設定状態であることが分かる表示名とアクセシビリティ名を提供する

### Requirement 3: タブ選択状態と再取得時の補正

**Objective:** As a 日付範囲、検索語、同期を使いながら履歴を見る利用者, I want タブ選択が現在の結果集合に追従してほしい, so that 条件変更後に存在しないディレクトリ表示へ取り残されない

#### Acceptance Criteria

1. When 利用者がタブを切り替える, the セッション一覧画面 shall 日付範囲と検索語を変更せず、追加の一覧取得要求を発生させずに表示対象を切り替える
2. When 日付範囲または検索語の適用により新しい成功状態のセッション集合が表示される, the セッション一覧画面 shall 新しいセッション集合からタブを再構築する
3. When 履歴同期後の再読み込みにより新しい成功状態のセッション集合が表示される, the セッション一覧画面 shall 新しいセッション集合からタブを再構築する
4. If 選択中の作業ディレクトリ別タブが新しいセッション集合に存在しない, then the セッション一覧画面 shall 選択中タブを `すべて` に補正する
5. While セッション一覧が loading、empty、または error 状態である, the セッション一覧画面 shall 既存の状態表示契約を維持し、成功状態のセッションタブを表示しない

### Requirement 4: タブ操作とアクセシビリティ

**Objective:** As a キーボードまたは支援技術を利用する利用者, I want 作業ディレクトリ別タブを一貫したタブ UI として操作したい, so that マウス操作に依存せず一覧を切り替えられる

#### Acceptance Criteria

1. When 利用者がクリックまたはタップでタブを選択する, the セッション一覧画面 shall 選択されたタブに対応するセッション一覧を表示する
2. When キーボードフォーカスがタブ一覧内にあり利用者が左右キーを押す, the セッション一覧画面 shall 隣接するタブへ選択を移動する
3. When タブが選択されている, the セッション一覧画面 shall 選択状態を視覚的に示し、支援技術にも選択状態を伝える
4. The セッション一覧画面 shall タブ一覧と各タブに、対象範囲と件数が分かるアクセシビリティ名を提供する
5. The セッション一覧画面 shall 長いパスや多いタブが存在しても、ページ全体の横スクロールやテキストの重なりを発生させない

### Requirement 5: 検索結果との連動と案内文の更新

**Objective:** As a 検索語で絞り込んだ履歴を確認する利用者, I want 検索結果内でさらに作業ディレクトリを切り替えたい, so that 検索条件とディレクトリの両方で目的のセッションに近づける

#### Acceptance Criteria

1. When 検索語が適用された成功状態のセッション集合が表示される, the セッション一覧画面 shall その検索結果集合だけから作業ディレクトリ別タブを構築する
2. When 検索語が適用された状態で `すべて` タブが選択される, the セッション一覧画面 shall 検索結果の全セッションを既存の一覧順で表示する
3. When 検索語が適用された状態で作業ディレクトリ別タブが選択される, the セッション一覧画面 shall 検索結果のうち該当作業ディレクトリのセッションだけを表示する
4. When 検索フォームの補助説明が表示される, the セッション一覧画面 shall 実行ディレクトリを検索対象として案内せず、作業ディレクトリは一覧タブで切り替える文脈に合わせた説明を表示する
5. If 検索条件に一致するセッションが存在しない, then the セッション一覧画面 shall 既存の検索 empty 表示を維持し、空のタブ一覧を表示しない

### Requirement 6: 既存契約とスコープ境界の維持

**Objective:** As a ローカル履歴アプリの利用者, I want 既存の一覧取得、検索、同期の振る舞いが変わらないままタブだけが追加されてほしい, so that これまでの絞り込みや同期操作を同じ期待で使い続けられる

#### Acceptance Criteria

1. The セッション一覧機能 shall 既存の日付範囲、検索語、同期後 refresh、loading、empty、error の利用者向け契約を維持する
2. The セッション一覧機能 shall 既存の一覧 API response shape を変更せずに作業ディレクトリ別タブを提供する
3. The セッション一覧機能 shall 作業ディレクトリ別タブのために directory summary endpoint、server-side aggregation、または pagination を要求しない
4. Where セッション一覧 API response に `work_context.cwd` が含まれる, the セッション一覧機能 shall その値を作業ディレクトリ別タブの唯一の cwd 根拠として扱う
5. If 既存 backend が検索語に対して `cwd` を含む検索互換性を維持している, then the セッション一覧機能 shall 返却済みセッション集合を権威として扱い、frontend 側で検索契約を変更しない
