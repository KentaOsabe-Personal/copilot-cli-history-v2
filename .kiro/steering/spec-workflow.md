# Spec 運用方針

updated_at: 2026-05-04

## 基本方針

このリポジトリでは、**spec はその時点の判断と境界を残す断面**として扱います。  
既存 spec をあとから継ぎ足して最新仕様の寄せ集めにするより、必要なら新しい spec を切り、軽微な変更は直接実装で扱う方針です。

## 判断ルール

### 1. 既存 spec は履歴として残す

requirements / design / tasks がまとまった spec は、その時点のスコープ判断を残す記録です。  
後続の小変更に合わせて既存 spec を書き換え続ける運用は避けます。

### 2. 新しい責務境界があるなら新規 spec を作る

変更内容が既存 spec に近くても、独立したレビュー境界や責務を持つなら新しい spec を作ります。  
「同じ画面に触る」だけでは既存 spec 更新の理由にせず、境界が独立しているかで判断します。

### 3. 軽微変更は spec なしで直接実装する

小さなバグ修正、既定値調整、表示ノイズの軽減、局所的な refactor のように、既存責務の範囲内で完結する変更は直接実装を優先します。  
spec を増やすことで得られる設計価値が薄い場合は、履歴を壊さないよう既存 spec を触らずに進めます。

### 4. discovery では「既存 spec 更新」を安易に選ばない

spec 運用上の第一候補は「新規 spec」または「直接実装」です。  
既存 spec 更新を提案するのは、同じ責務境界を継続して育てる明確な理由がある場合に限ります。

## 運用上のメモ

- steering は project-wide な判断基準を残す
- specs は feature ごとの要件・設計・タスクを残す
- 直接実装を選ぶ場合でも、テストと README など必要な周辺更新は省略しない

---
_spec の履歴性を守りつつ、軽微変更で仕様書メンテナンスが過剰にならないための運用ルール_
